# PDF Report Replan — Design & Implementation Guide

**Supersedes**, for layout/formatting purposes, the original visual spec in [PDF_EXPORT_PLAN.md](PDF_EXPORT_PLAN.md) (that document's architecture — cover+body two-pass print, `pdf-lib` merge, automatic/once/immutable generation, "PDF generation never fails the run" — is unchanged and still authoritative; only the cover content and the Full Conversation section's presentation are being replaced).

---

## Understanding Summary

- **What**: a full replan of the PDF's cover page, running header, and Full Conversation section, moving from today's Space-name-led cover + per-agent bordered cards to a formal/corporate-styled report whose conversation section is a flowing, AI-narrated story instead of raw cards.
- **Why**: general polish so the PDF reads as professional enough to forward to an institution or agency — not a response to a defect.
- **Who**: the user, who generates these reports and hands them off externally.
- **Key constraint**: the new AI-written narrative may reorganize/narrate freely, but anything presented as a direct quote from an agent must be **exactly what that agent said** — mechanically verified, not just prompted for.
- **Non-goals**: rewriting the Final Answer's actual text (it stays verbatim, always); changing the two-pass print/merge architecture; adding a report reference number or confidentiality marking to the cover; per-Space narrative model selection (it's global).

## Assumptions (carried over from the brainstorm, restated for the record)

1. Exact colors/fonts for "formal/corporate" are proposed below as concrete defaults, easily tunable later.
2. The narrative pass runs on whatever transcript exists even for a stopped/failed run.
3. One AI call per report (not per-turn), covering both the Key Points summary and the full narrative.
4. If no narrative model is configured in Settings, the AI step is skipped entirely — zero added latency, today's layout is used.
5. Report generation stays "automatic, once, immutable" — no regeneration/editing UI.

| 5 | Running header shows a short version of the problem + date, replacing Space name | Keep Space name; use preset name | Consistent with the cover no longer featuring the Space name |
| 6 | Footer content unchanged (brand + page count), only restyled | Add date to footer | User's choice — keep it simple |
| 7 | Final Answer box shows the run's `finalAnswer` **verbatim, always** — never AI-rewritten | Let the AI rewrite the conclusion too | The final answer is the actual answer to the user's problem; accuracy here is non-negotiable, more so than in supporting narrative |
| 8 | The narrative call *also* produces a short "Key Points" bullet list, shown above the verbatim answer inside the same highlighted box, only when the AI pass succeeds | Always show bullets (needs AI); never show bullets | Resolves the user's "point-wise summary" want without ever touching the verbatim answer — bullets are clearly the AI's own condensation, not a quote, so they need no verbatim check |
| 9 | Full Conversation section fully replaced by AI narrative (when available), not shown alongside cards | Show both narrative and cards | User's explicit choice — single clean section, not two versions of the same content |

| 11 | Quotes are tagged by the agent's **role** (e.g. "White Hat"), not its internal `name` field | Tag by `name` | Preserves the original PDF decision that cards show role, not internal naming — more meaningful to an external reader |
| 12 | Tool results are excluded from what the AI sees (only tool name + input), same as today's cards | Include tool results | Consistency with the existing decision to keep tool result data out of the PDF; also reduces prompt size |
| 13 | Narrative model is a **global** Settings field, not per-Space | Per-Space setting | User's choice — one setting, matches where other report-related settings (reports folder) already live |
| 14 | If the transcript text exceeds ~12,000 characters, skip the AI pass immediately without calling the model | Send it anyway and let it fail | Mirrors the existing `MAX_RESPONSE_CHARS` precedent in `WebhookClient.ts` (protecting model context); avoids a doomed, slow call |
| 15 | A failed/skipped narrative pass is logged (`console.error`) but does **not** emit a visible run/System event | Surface it as a System note like PDF-write failures do | The end result (a complete, correct PDF) is unaffected from the user's perspective — this is an expected, silent fallback, not an error state worth flagging in the transcript |

## Design

### 1. Cover page (`ReportRenderer.ts`, `coverHtml`)

```
[Problem statement — large serif title]
Methodology: Six Thinking Hats

"..." <- no, the problem IS the title now, not quoted separately
[date]
[disclosure line, small, bottom]
```

Concretely: `<h1>` becomes `escapeHtml(run.problem)` (previously the quoted `.problem` block; that block is removed since the problem *is* the title now — no need to show it twice). The `Methodology:` line only appears when `space.presetId` resolves to a preset. Space name and the status line are removed. Disclosure line unchanged.

**New cover typography** (proposed defaults, all within the existing embedded `<style>` block in `coverHtml`):
- Headings: `Georgia, 'Times New Roman', serif` (a classic, universally available serif — reads as formal, unlike the current all-sans design)
- Body/labels: keep the existing `system-ui` sans stack for the methodology line, date, disclosure
- Primary color: navy `#1e3a5f` replacing the current slate `#1e293b`; muted label color `#5b6b7f` replacing `#64748b`

### 2. Running header/footer (`PdfWriter.ts`)

`writeRunPdf`'s header currently takes `spaceName`. It changes to take the **problem**, truncated to ~70 characters with an ellipsis if longer (so a long problem statement doesn't overflow the header). Footer is unchanged in content — only its font/colors move to match the new navy palette. This means `writeRunPdf`'s signature changes from `(report, outPath, spaceName, dateStr)` to `(report, outPath, problem, dateStr)`, and its one caller (`RunManager.ts`) updates accordingly.

### 3. Final Answer section (`ReportRenderer.ts`, `bodyHtml`)

Always renders `run.finalAnswer` verbatim (via the existing `renderSafeMarkdown`) inside a new highlighted box style (`.answer-box` — distinct from the existing `.result-box`/`.stopped-box` used for failed/stopped states, which are unchanged). When a valid `narrative` result is available, a "Key Points" subheading + bullet list (from `narrative.keyPoints`) renders **above** the verbatim answer, inside the same box. When no narrative is available, the box shows just the verbatim answer, exactly as today's content (just newly boxed).

### 4. Full Conversation → narrative (`ReportRenderer.ts`, `bodyHtml`)

When `input.narrative` is present: render `narrative.narrativeMarkdown` through `renderSafeMarkdown`, after converting every `<quote agent="Role">text</quote>` into a markdown blockquote with attribution (`> "text"\n> — Role`) — this conversion happens **after** validation (which happened in `NarrativeGenerator`), so by the time this text reaches `renderSafeMarkdown` it's plain markdown, going through the exact same safe-rendering path as everything else (no new HTML-injection surface). Heading stays "Full Conversation" for continuity.

When `input.narrative` is absent (no model configured, call failed, or a quote failed validation): render **exactly today's code** — the per-agent `.card` loop over events, completely unchanged. This is a straightforward `if/else` at the top of the conversation-rendering block, not a rewrite of the existing loop.

### 5. New module: `packages/core/src/report/NarrativeGenerator.ts`

The heart of this replan. A single exported async function:

```ts
export interface NarrativeResult {
  keyPoints: string[];        // AI's own condensation — no verbatim check needed
  narrativeMarkdown: string;  // validated; <quote agent="Role">text</quote> tags still embedded
}

export async function generateNarrative(
  input: RunReportInput,
  lmClient: LmStudioClient,
  model: string
): Promise<NarrativeResult | undefined> {
  const transcript = buildPlainTranscript(input);           // events -> readable plain text
  if (transcript.length > MAX_TRANSCRIPT_CHARS) return undefined;  // Decision #14

  let response: ChatResponse;
  try {
    response = await lmClient.chat(
      { model, messages: buildNarrativePrompt(input, transcript), temperature: 0.3 },
      () => {},                       // no live streaming needed for a background pass
      undefined,
      { overallTimeoutMs: 180_000 }   // bounded; this is a background pass, not agent turns
    );
  } catch {
    return undefined;                 // LM Studio unreachable / stall timeout / etc.
  }

  const parsed = parseNarrativeResponse(response.message.content);
  if (!parsed) return undefined;      // missing/malformed [KEY_POINTS] or [NARRATIVE] blocks

  if (!allQuotesVerified(parsed.narrativeMarkdown, input.agents, input.events)) {
    return undefined;                 // Decision #10 — any single bad quote kills the whole pass
  }

  return parsed;
}
```

**`buildPlainTranscript`** reuses the same turn-grouping logic already established in `ReportRenderer.ts`'s per-agent loop (`RoundStart` opens a turn keyed by `agentId`, `AgentMessage` fills its body, `ToolCall` records name+input, `ToolResult` is skipped per Decision #12, `System` events become standalone notes). Output looks like:

```
Problem: How to control the Indian population?

[Turn] White Hat: <full agent message content>
  (used tool "webhook__Search_web" — query: "India population policy 2026")
[Turn] Black Hat: <full agent message content>
[System note] Max rounds reached without a declared answer; synthesizing a best-effort answer.
```

**`buildNarrativePrompt`** is a system+user message pair instructing the model: write a short bullet list of key points, then a flowing narrative of the discussion, using the exact tag contract below, and explicitly stating which agent (by role) is the orchestrator/manager so it can be narrated as coordinating the discussion, matching the user's own example ("Manager decides..."). Expected response shape:

```
[KEY_POINTS]
- Point one
- Point two
[/KEY_POINTS]

[NARRATIVE]
White Hat opened by noting <quote agent="White Hat">the current data shows...</quote>. Black Hat then raised concerns that <quote agent="Black Hat">this risks...</quote>. As the discussion's coordinator, Blue Hat weighed both perspectives and...
[/NARRATIVE]
```

**`parseNarrativeResponse`** — regex-extracts the two tagged blocks (same tolerant-tag philosophy as `parseTaskAssignments`/`extractFinalAnswer`); returns `undefined` if either block is missing.

**`allQuotesVerified`** — extracts every `<quote agent="Role">text</quote>` via regex, and for each: finds the agent(s) whose `role` matches (case-insensitive) the tag's `agent` attribute, concatenates all of that agent's real `AgentMessage` content from `events`, normalizes whitespace (trim + collapse runs of whitespace to a single space) on both sides, and checks the quoted text is an exact substring. **Any unmatched agent role, or any quote that isn't a genuine substring, fails the whole batch.** A minimum quote length (~15 characters) guards against trivially-matching near-empty quotes.

### 6. Wiring — `RunManager.ts`

Before building the report, `RunManager` now:

```ts
const narrativeModel = this.getReportNarrativeModel();
const narrative = narrativeModel
  ? await generateNarrative({ run: finalRun, space, agents, events }, this.getLmStudioClient(), narrativeModel).catch(() => undefined)
  : undefined;
const report = renderRunReport({ run: finalRun, space, agents, events, narrative });
```

`getReportNarrativeModel: () => string | undefined` is a new constructor dependency, mirroring the existing `getReportsFolder` pattern, wired in `main/index.ts` from `SettingsStore`.

### 7. New setting — `SettingsStore.ts` + `SettingsScreen.tsx`

`Settings` interface gains `reportNarrativeModel?: string`, defaulting to `undefined` (feature off). `SettingsScreen.tsx` gets a new "PDF report narrative model" dropdown, reusing the same model-list-fetch pattern already used for a Space's default model — with a hint explaining what it does and that leaving it unset keeps today's plain layout.

### 8. `RunReportInput` / `ReportRenderer.ts` signature change

```ts
export interface RunReportInput {
  run: Run;
  space: Space;
  agents: Agent[];
  events: RunEvent[];
  narrative?: NarrativeResult;   // NEW, optional
}
```

`ReportRenderer.ts` itself stays a **pure, synchronous, Electron-free function** — it never calls an LLM. It only decides, based on whether `narrative` is present and was already validated upstream, which of the two rendering paths (AI narrative vs. today's cards) to take. This keeps the existing "core stays Electron-free" and "test-first for the engine" rules fully intact — `NarrativeGenerator` is the only new piece that talks to an LLM, and it's independently testable with a fake `LmStudioClient`, exactly like `RunOrchestrator` already is.

## Test Plan

**`NarrativeGenerator.test.ts`** (new):
- Well-formed response with genuinely matching quotes → returns a populated `NarrativeResult`.
- A quote that doesn't match the real transcript (even slightly) → returns `undefined`.
- Missing `[KEY_POINTS]` or `[NARRATIVE]` block → returns `undefined`.
- `chat()` throws/times out → caught, returns `undefined`, never propagates.
- Transcript exceeds the character cap → returns `undefined` **without calling `chat()` at all** (assert the spy was never invoked, same style as the existing "halted before any agent ran" test in `engine.test.ts`).
- Quote tagged with an agent role that doesn't exist in `agents` → validation failure.

**`ReportRenderer.test.ts`** (update):
- Cover: `<h1>` contains the problem, does **not** contain the Space name anywhere; contains "Methodology: X" when `presetId` is set; no status line for a failed/stopped run.
- Body, `narrative` provided and valid: contains the rendered narrative text and quote attribution; does **not** contain any `.card` element.
- Body, `narrative` omitted: renders **exactly today's** per-agent cards (regression guard — keep the existing assertions).
- Final Answer box: shows Key Points bullets when narrative present; shows only the verbatim answer when absent; verbatim answer text is identical in both cases.

**`PdfWriter.test.ts`**: verify/update for the `spaceName` → `problem` (truncated) header parameter change.

**`RunManager.test.ts`**: new test wiring `getReportNarrativeModel` returning a model + a `chat` mock producing a valid narrative response, asserting the report passed to `writePdf` reflects it; existing tests continue to pass unchanged with the dependency defaulted to `undefined`.

**`SettingsStore.test.ts`**: round-trip `reportNarrativeModel` (get/set), consistent with existing settings tests.

Full `npm run verify` after each step.

## Implementation Order

1. **Cover + header/footer redesign** (`ReportRenderer.ts` cover, `PdfWriter.ts` header) — no AI dependency, immediately testable, delivers visible polish on its own.
2. **`NarrativeGenerator.ts`** + its tests — build and prove the generation+validation logic in isolation before touching the renderer.
3. **`ReportRenderer.ts` body changes** (Final Answer box, narrative vs. cards branch) + updated tests.
4. **Settings field + UI** (`SettingsStore.ts`, `SettingsScreen.tsx`).
5. **`RunManager.ts` wiring** — the last step, since it's the integration point pulling everything together.

Each step: `npm run verify`, then a real run in the app to eyeball the actual PDF.

## Acceptance Criteria

- A generated PDF's cover shows the problem as the title, "Methodology: X" when applicable, the date, and the disclosure line — no Space name, no status line.
- The running header shows a short version of the problem + date; footer is unchanged in content, restyled.
- With no narrative model configured: the PDF is identical in structure to today's (cards, plain Final Answer) except for the new cover/header/footer styling.
- With a narrative model configured and a healthy response: the Full Conversation section reads as flowing narrative prose with attributed quotes, and every quoted sentence is byte-for-byte verifiable against the real transcript.
- Any single fabricated/altered quote in the AI's response results in the **entire** classic layout being used instead — never a partially-trusted narrative.
- `npm run verify` green after each step.
